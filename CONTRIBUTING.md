# 贡献指南（Contributing Guide）

感谢你对 **g2rain-mybatis-extensions** 的关注与贡献意愿！
为了让协作更顺畅、更高效，请在参与项目前阅读本指南。

---

## 1. 我可以如何贡献？

无论你是刚接触本项目，还是已经深入使用，都有很多方式可以参与：

- **使用反馈**：在实际项目中使用本项目，并反馈遇到的问题与需求。
- **文档改进**：修正文档中的错别字、过期内容、示例不清晰等问题。
- **Bug 修复**：修复现有 Issue 中标记的 Bug，或你自己发现的问题。
- **新特性**：为项目提出或实现新功能（建议先通过 Issue 讨论）。
- **示例与教程**：补充 Demo、最佳实践、集成案例等。

---

## 2. 提交 Issue 前请先检查

在创建 Issue 之前，请先：

1. 搜索现有 Issue，避免重复提问。
2. 尽量确认问题属于项目本身，而不是环境或依赖配置问题。

创建 Issue 时建议包含：

- 使用的版本（例如：`g2rain-mybatis-extensions@1.0.0`）。
- 运行环境（JDK 版本、MyBatis 版本、Spring Boot 版本、操作系统等）。
- 复现步骤（最好附上最小可复现示例）。
- 实际行为 vs 期望行为。

---

## 3. 提交 Pull Request（PR）流程

1. **Fork 仓库**
   在 GitHub 上 Fork 本仓库，并 Clone 到本地。

2. **创建分支**
   建议从 `main`（或维护者指定的开发分支）拉一个新分支：

```bash
git checkout -b feat/xxx
# 或
git checkout -b fix/issue-123
```

3. **本地构建**

```bash
mvn -B clean test
```

4. **保持提交粒度清晰**
   - 一个 PR 尽量专注于一个问题或一个特性。
   - Commit 信息建议遵循简单的约定式（如 `fix: xxx`、`feat: yyy`、`docs: update zzz`）。

5. **提交 PR**
   - 填写清晰的标题和描述。
   - 链接相关 Issue（例如：`Close #123`）。
   - 说明改动动机、设计思路以及可能的影响范围。

6. **Review 与修改**
   Maintainer 会进行 Review，并可能提出修改建议；
   请根据反馈更新 PR，直到达成共识。

---

## 4. 代码风格与规范

- 保持与现有代码风格一致。
- 避免在无必要时大规模重排代码，以免影响 Review。
- 提交前尽量执行：

```bash
mvn -B clean test
```

---

## 5. 行为准则

所有参与者都应遵守本项目的 [行为准则](CODE_OF_CONDUCT.md)，保持友好、专业、尊重的交流氛围。
